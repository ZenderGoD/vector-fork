# Welcome to AIKP Documentation

This documentation provides a comprehensive guide to the AIKP project. Whether you are a developer, a designer, or a product manager, you will find the information you need to get started and contribute to the project.

## Documentation Sections

This documentation is divided into the following sections. For a new developer implementing a feature, these guides provide the necessary rules and instructions.

### 1. [Getting Started](./getting-started/01-local-setup.md)

This section covers everything you need to get the project running on your local machine. It includes initial setup, environment variables, database configuration, and common commands.

### 2. [Product](./product/01-overview.md)

This section contains information about the product itself. It includes an overview of what AIKP is, its key features, and the different user roles within the system.

### 3. [Architecture](./architecture/01-stack.md)

This section offers a deep dive into the project's technical architecture. You'll find details on the tech stack, project structure, the authentication and permissions system, and how file storage is handled.

### 4. [Development](./development/01-coding-style.md)

This section contains guides and conventions for developers. It covers coding style, how to make database changes, API and routing conventions, type safety rules, and how to use the component library.

### 5. [Design](./design/01-design-system.md)

This section outlines the project's design system and UX guidelines. It defines the visual style, component usage, and standard design patterns to ensure a consistent user experience.

### 6. [Architectural Decision Records (ADR)](./adr/001-service-layer-implementation-plan.md)

This section contains a log of significant architectural decisions. ADRs provide context and justification for why the system is built the way it is.

---

## Contributing

We follow the conventions and guides detailed in this documentation. Please ensure `pnpm lint` passes and keep commits small and logical.

Feel free to open issues or PRs – feedback is welcome!
